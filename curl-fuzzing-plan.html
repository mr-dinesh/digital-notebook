curl CVE-Guided
Fuzzing Plan

// AFL++ · Ghidra · Volatility · Hallucination-Resistant Workflow

⚠ ANTI-SLOP CONTRACT Every AI-suggested finding must be verified against actual source code, crash trace, or patch diff before being treated as real. Claude is a triage assistant, not an authority. Ground truth lives in the code.
PHASE 01 Environment Setup & Baseline Validation
1
Clone curl and build with sanitizers

Build from source with -fsanitize=address,undefined. This is your bug detector — without it, crashes are silent.

2
Install AFL++ with instrumentation

Use afl-clang-fast to compile curl. Coverage-guided fuzzing only works if the binary is instrumented correctly. Verify with afl-showmap.

3
Rediscover a known CVE first — validate your setup

Checkout the vulnerable commit for CVE-2023-38545. Run your harness. If you can reproduce the known crash, your environment is trustworthy. If you can't, fix setup before proceeding. Do not skip this step.

PHASE 02 CVE Pattern Mapping
1
Pull patch diffs for 3 anchor CVEs

CVE-2023-38545 (SOCKS5 heap overflow), CVE-2022-32221 (POST redirect), CVE-2023-46218 (cookie handling). Read the actual diff — not summaries.

2
Extract root cause pattern per CVE

For each: what assumption was wrong? What input triggered it? What memory operation failed? Write this in one sentence per CVE in your own words.

3
Use Claude to suggest adjacent functions — then verify manually

Ask Claude which functions share the same pattern. Then open lib/url.c and related files and read the suggested functions yourself. Discard any suggestion you cannot verify in source.

PHASE 03 Targeted Harness Construction
1
Write one focused harness per target function

Do not fuzz main(). Write harnesses that call the specific vulnerable-pattern function directly with AFL-controlled input. Smaller attack surface = faster meaningful coverage.

2
Build Ghidra-assisted dictionary

Run Ghidra headless on the curl binary. Extract string constants, magic bytes, comparison literals. Feed as -x dict to AFL++. Now your mutations are semantically informed.

3
Seed corpus from real traffic

Capture legitimate curl request/response data as seeds. AFL mutates from working inputs faster than from random bytes.

PHASE 04 Crash Triage Pipeline
1
Reproduce every crash manually first

Run the crash input against the instrumented binary directly. Confirm it crashes reproducibly. If it doesn't reproduce, discard it — do not analyze phantoms.

2
Deduplicate by stack trace, not by count

AFL will find hundreds of inputs triggering the same bug. Group by crash address and call stack. One unique stack = one potential bug.

3
Feed crash + source context to Claude for triage

Provide the sanitizer output, the crashing input, and the relevant source function. Ask Claude to explain the crash. Then verify that explanation against the source yourself before recording it as a finding.

4
Cross-reference against OSS-Fuzz and existing CVEs

Before claiming a new finding, check OSS-Fuzz issue tracker and NVD. curl is heavily fuzzed. A "new" crash that's already known is a learning experience, not a failure.

PHASE 05 Volatility Integration (Memory Track)
1
Start with a known memory forensics challenge

Use CyberDefenders or MemLabs — images with known answers. Build confidence in your Volatility workflow before using it on original targets.

2
Connect tracks: Volatility anomaly → Ghidra RE

Flag a suspicious process in Volatility. Dump the PE. Feed to Ghidra for static analysis. This is the intersection point — and the workflow worth automating as a project entry.

// HALLUCINATION_CONTROLS

Where AI goes wrong in this workflow — and how to catch it

Risk 01

Fabricated function analysis

Claude confidently describes what a function does — but hasn't seen the actual source, only inferred from context.

✓ Open the file. Read the function. Every time.
Risk 02

Plausible but wrong adjacency

Claude suggests function X has the same vulnerability pattern as the CVE. It sounds right but the code doesn't support it.

✓ Paste the actual function into Claude. Don't let it infer from a name.
Risk 03

Crash misinterpretation

Claude explains a crash trace convincingly but identifies the wrong root cause — missing a one-level-up allocation failure.

✓ Always provide full sanitizer output + source. Ask Claude to show its reasoning step by step.
Risk 04

Already-known bug presented as new

Claude doesn't know OSS-Fuzz's issue tracker. A finding it calls novel may be filed and patched already.

✓ Check OSS-Fuzz issues and NVD before claiming any finding. Takes 5 minutes.
